Por que testar carga?

Toda Black Friday, sistemas que passam o ano aparentemente saudáveis caem em horas:

  • Conexões ao banco esgotam: filas, timeouts, 503
  • Memory leaks que demoram horas para aparecer estouram a JVM/Node
  • Thread/event loop starvation deixa o app vivo mas inerte
  • Caches frios e conexões TLS recém-abertas multiplicam latência

Como descobrir antes do dia D — e não junto com os clientes?

Tipos de teste de performance

Load

Carga esperada sustentada. Mede regime estacionário.

Stress

Sobe até quebrar. Descobre limite máximo.

Spike

Pico abrupto. Simula flash sale, viral, deploy.

Soak

Carga moderada por horas. Detecta vazamento, fadiga.

Anatomia da carga: ramp-up · steady · ramp-down

Ramp-up — sobe gradual.
Por quê? Aquece caches, JIT, conexões, evita falso positivo de cold-start.

Steady — carga constante.
Por quê? Aqui se medem P95/P99 confiáveis — o sistema atingiu regime estacionário.

Ramp-down — desce gradual.
Por quê? Detecta recursos não liberados (conexões abertas, memória não devolvida).

Métrica 1 — Throughput (RPS)

Definição: requisições processadas por segundo.

Fórmula: requests / segundos

Quando importa: capacidade total — quantos clientes simultâneos seu sistema atende?

Throughput é capacidade; latência é experiência.

Métrica 2 — Latência: P50 · P95 · P99

P50 (mediana) — metade das requisições é mais rápida que isso. Não conta a história inteira.

P95 — 95% terminam em até esse tempo. 5% sentem mais.

P99 — 1 em 100 requisições. O usuário que tweeta sobre a sua app.

A média esconde a cauda. O P99 conta a verdade da pior experiência real.

Métrica 3 — Error Rate

Percentual de requisições que falharam. O que conta como falha?

  • HTTP 5xx (server error)
  • Timeout (cliente desistiu antes da resposta)
  • Body inválido ou inesperado (validar com check() no k6)
  • HTTP 4xx em fluxos onde não deveriam aparecer
FaixaSignificadoAção
< 0,1%SaudávelContinuar monitorando
0,1% – 1%AtençãoInvestigar erros, identificar padrão
1% – 5%CríticoProvavelmente saturação ou bug; agir hoje
> 5%Degradação abertaRollback, escalar, abrir incidente

No k6: http_req_failed — threshold típico: ['rate<0.01']

Métrica 4 — Saturação

Definição (Brendan Gregg, USE Method):
Demanda além da capacidade que o recurso pode atender no momento.

Recursos típicos a observar:

  • CPU (cores ocupadas)
  • Memória (RAM, heap)
  • Event loop lag (Node)
  • Connection pool do DB
  • Threads (em runtimes thread-based)

k6 mede do lado do cliente. Saturação real é vista no servidor (APM, Prometheus, top, etc.).

Como ler um relatório do k6

http_reqs — total + RPS médio. Sua taxa de produção.

http_req_duration com p(95) / p(99) — latência percentilada. Olhe aqui antes da média.

http_req_failed — taxa de falha. Threshold típico: <1%.

iteration_duration — tempo total de uma iteração (req + sleep). Útil para entender ritmo do VU.

vus / vus_max — quantos usuários virtuais ativos. Confirma que o ramp-up rodou.

Hot Path — onde otimizar primeiro

Hot path = caminho de código que recebe a maior parte do tráfego.

Como identificar:

  • APM / observabilidade (NewRelic, Datadog, Application Insights)
  • Logs agregados por endpoint
  • Pareto: 20% dos endpoints atendem 80% das requisições

Ganhar 1% de performance no hot path rende mais que 50% nos endpoints frios.

k6 — instalação passo a passo

k6 é a ferramenta open-source da Grafana Labs para testes de performance. Roda como binário nativo (Go) — não precisa de Node nem JVM. Testes são escritos em JavaScript.

Windows

winget install k6 --source winget

Alternativa: choco install k6

macOS

brew install k6

Requer Homebrew (brew.sh)

Linux (Debian/Ubuntu)

sudo gpg --keyserver hkp://keyserver.ubuntu.com:80 \
  --recv-keys C5AD17C747E3415A...
echo "deb [signed-by=...] https://dl.k6.io/deb stable main" \
  | sudo tee /etc/apt/sources.list.d/k6.list
sudo apt update && sudo apt install k6
Validação: 
k6 version
→ deve imprimir k6 v0.x.x
