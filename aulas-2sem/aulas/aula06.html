Por que usar cache? Os números

OperaçãoLatênciaComparação

Na prática: Redis é ~100x mais rápido que uma query PostgreSQL típica. Em um sistema com 10.000 req/s, cache pode reduzir a carga do banco em 90%+ para dados lidos com frequência.

EstratégiaQuem lê do banco?Quem escreve no cache?Melhor para

As estratégias de leitura (Cache-Aside, Read-Through) e escrita (Write-Through, Write-Behind) podem ser combinadas. Ex: Cache-Aside para leitura + Write-Through para escrita.

Cache-Aside (Lazy Loading)

VantagensDesvantagens
Só carrega no cache o que realmente é lido Primeiro request sempre tem latência total (cold start)
Cache falhar não derruba a aplicação Dados podem ficar stale entre TTL e atualização no banco

Caso real: Netflix usa Cache-Aside para catálogo de filmes. ~90% dos requests são servidos pelo cache. Dados atualizados a cada 15-30 min via TTL — aceitável porque metadados de filmes mudam raramente.

Read-Through

VantagensDesvantagens
Código da aplicação mais limpo — não gerencia cache Precisa de um cache provider que suporte loaders
Centraliza a lógica de carregamento no provider Debugging mais difícil (lógica escondida na camada de cache)

Na prática: bibliotecas como Caffeine (Java) e NCache (.NET) implementam Read-Through nativamente. Você configura um loader e a lib cuida do resto. Redis puro não suporta Read-Through — precisa de uma camada wrapper na aplicação.

Write-Through

VantagensDesvantagens
Cache sempre consistente com o banco Latência de escrita ~2x (escreve em dois lugares)
Leituras subsequentes sempre HIT Pode cachear dados que nunca serão lidos (desperdício de memória)

Caso real: DynamoDB Accelerator (DAX) da AWS usa Write-Through. Toda escrita no DynamoDB passa pelo DAX automaticamente, garantindo que leituras sempre retornem dados atualizados com latência de microsegundos.

Write-Behind (Write-Back)

VantagensDesvantagens
Latência de escrita mínima (só cache) Risco de perda de dados se o cache falhar antes do sync
Permite agrupar escritas em batch (reduz carga no banco) Complexidade de implementação (fila, retries, dead-letter)

Caso real: Sistemas de analytics e contadores (views, likes, page hits) usam Write-Behind. O Facebook acumula contadores de likes em memória e faz flush para o banco a cada poucos segundos — perder um ou dois likes é aceitável, mas a latência de escrita precisa ser mínima.

Redis: Políticas de Eviction (maxmemory-policy)

PolicyComportamento
noevictionRetorna erro quando memória cheia (padrão)
allkeys-lruRemove chave menos acessada recentemente (recomendado para cache)
volatile-lruLRU apenas em chaves com TTL definido
allkeys-lfuRemove chave com menor frequência de acesso
volatile-lfuLFU apenas em chaves com TTL definido
volatile-ttlRemove chave com menor TTL restante
 
# Configurar limite de memória e política
CONFIG SET maxmemory 256mb
CONFIG SET maxmemory-policy allkeys-lru

Regra prática: para cache puro, use allkeys-lru. Para mix de cache + dados persistentes, use volatile-lru (só remove chaves com TTL).

Redis como Cache Distribuído

  • Armazenamento chave-valor em memória, escrito em C
  • Suporta múltiplas estruturas: strings, hashes, listas, sets, sorted sets
  • Alta disponibilidade com replicação (master-replica) e persistência opcional
  • Suporte a clusters para escalabilidade horizontal (particionamento automático)
  • Ideal para cache distribuído em arquiteturas de microsserviços

Redis: Números de Performance

Latência média< 1 ms para operações simples

Por que single-threaded? Evita overhead de locks e context switching. O gargalo geralmente é a rede, não a CPU. Redis 7+ usa threads auxiliares para I/O de rede, mantendo a execução de comandos single-threaded.

Redis Keys: Padrão e Convenções

Formato recomendado: objeto:id:atributo
Exemplos: user:123:profile, session:abc-def, cache:product:456
Separador padrão: : (dois-pontos)
Convenção universal no ecossistema Redis. Facilita organização e busca.
Prefixos por serviço (microsserviços):
order-svc:order:789, auth-svc:session:abc — evita colisão entre serviços
Evitar:
Chaves > 1 KB, espaços, caracteres especiais. Chaves curtas = menos memória.
# Boas práticas de nomeação
SET user:123:name "João"
SET user:123:email "joao@email.com"
SET session:abc-def-ghi '{"userId":"123"}'
SET cache:product:456 '{"nome":"Notebook","preco":4500}'

Estruturas de Dados Redis — Visão Geral

Redis não é apenas chave-valor de strings. Ele oferece 5 estruturas nativas, cada uma otimizada para um padrão de acesso diferente:

TipoAnalogiaComplexidade (maioria)Melhor para
StringVariável simplesO(1)Cache, contadores, flags
HashObjeto / MapO(1) por campoPerfis, sessões, config
ListArray / FilaO(1) push/popFilas, timeline, histórico
SetConjunto matemáticoO(1) add/removeTags, permissões, unicidade
Sorted SetSet com notaO(log N)Rankings, leaderboards, agendamentos

Regra prática: escolha a estrutura pelo padrão de acesso, não pelo dado. Se você serializa JSON em String e depois faz parse para ler um campo, provavelmente deveria estar usando Hash.

06 - Cache com Redis

String

A estrutura mais simples e versátil do Redis

Armazena texto, números ou dados binários (até 512 MB). É a base do cache simples e de contadores atômicos.

# Cache simples (JSON serializado)
SET cache:product:42 '{"nome":"Notebook","preco":4500}' EX 600

# Contadores atômicos (thread-safe, sem race condition)
SET api:requests:today 0
INCR api:requests:today          # retorna 1
INCRBY api:requests:today 10     # retorna 11

# Flags e controles
SET feature:dark-mode:user:123 "enabled" EX 86400
SETNX lock:pedido:789 "worker-1"  # SET se NÃO existir (distributed lock!)

# Operações úteis
MSET user:1:name "Ana" user:1:email "ana@mail.com"   # SET múltiplo
MGET user:1:name user:1:email                         # GET múltiplo

Caso real: INCR é usado massivamente para rate limiting. A cada request, INCR api:rate:{ip} com TTL de 60s. Se ultrapassar o limite, retorna 429. Atômico, sem locks, ~100k ops/s.

Hash

Objetos com campos — como uma linha de banco no cache

Armazena pares campo-valor dentro de uma chave. Ideal para representar entidades sem serializar/deserializar JSON.

# Criar/atualizar campos de um usuário
HSET user:123 nome "Ana" email "ana@mail.com" idade 28 plano "premium"

# Ler um campo específico (sem trazer o objeto inteiro!)
HGET user:123 plano               # "premium"

# Ler todos os campos
HGETALL user:123                   # retorna todos os pares campo-valor

# Incrementar campo numérico
HINCRBY user:123 idade 1           # aniversário! agora 29

# Verificar se campo existe
HEXISTS user:123 telefone          # 0 (não existe)

# Remover campo
HDEL user:123 plano

Use Hash quando...

  • Precisa ler/atualizar campos individuais
  • Objetos com muitos campos (perfil, sessão, carrinho)
  • Quer economizar memória vs múltiplas Strings

Evite Hash quando...

  • O objeto inteiro é sempre lido/escrito junto (use String + JSON)
  • Precisa de TTL por campo (TTL é por chave, não por campo)

List

Listas ordenadas por inserção — filas e timelines

Lista duplamente encadeada. Inserção/remoção nas pontas em O(1). Ideal para filas (FIFO), pilhas (LIFO) e históricos limitados.

# Fila de tarefas (FIFO): produtor insere à esquerda, consumidor retira da direita
LPUSH fila:emails "msg-001" "msg-002" "msg-003"
RPOP fila:emails                    # "msg-001" (primeiro a entrar, primeiro a sair)

# Fila bloqueante (consumidor espera se fila vazia — ideal para workers)
BRPOP fila:emails 30               # bloqueia até 30s esperando item

# Timeline / histórico (últimas N ações)
LPUSH timeline:user:123 '{"acao":"login","ts":"2026-03-24T19:30:00"}'
LPUSH timeline:user:123 '{"acao":"compra","ts":"2026-03-24T19:45:00"}'
LTRIM timeline:user:123 0 99       # mantém apenas os últimos 100 itens
LRANGE timeline:user:123 0 9       # últimos 10 itens

# Verificar tamanho
LLEN fila:emails                    # quantos itens na fila

Caso real: O Sidekiq (Ruby) e o BullMQ (Node.js) usam Redis Lists como fila de jobs. LPUSH para enfileirar, BRPOP para consumir. Milhões de jobs/dia com latência de milissegundos.

Set

Coleção de valores únicos — operações de conjunto

Conjunto não-ordenado de strings únicas. Suporta operações matemáticas de conjuntos (união, interseção, diferença) em O(N).

# Tags de um post
SADD tags:post:42 "redis" "cache" "nosql" "performance"
SADD tags:post:77 "redis" "docker" "devops"

# Verificar se elemento existe (O(1))
SISMEMBER tags:post:42 "redis"     # 1 (existe)
SISMEMBER tags:post:42 "java"      # 0 (não existe)

# Operações de conjunto
SINTER tags:post:42 tags:post:77   # {"redis"} — tags em comum
SUNION tags:post:42 tags:post:77   # todas as tags únicas
SDIFF tags:post:42 tags:post:77    # {"cache","nosql","performance"} — só no 42

# Controle de sessões ativas
SADD online:users "user:123" "user:456" "user:789"
SCARD online:users                 # 3 (quantos online)
SREM online:users "user:456"       # usuário saiu

# Elemento aleatório (sorteios, A/B test)
SRANDMEMBER tags:post:42 2         # 2 tags aleatórias

Caso real: O Instagram usa Sets para armazenar quem curtiu cada foto. SISMEMBER responde instantaneamente "este usuário já curtiu?" e SCARD retorna a contagem total sem varrer todos os registros.

Sorted Set (ZSet)

Set com score — rankings, leaderboards e agendamentos

Cada membro tem um score numérico. Membros são únicos, ordenados automaticamente por score. Inserção e busca por rank em O(log N).

# Leaderboard de um jogo
ZADD ranking:game 1500 "player:ana" 1200 "player:bob" 1800 "player:carol"
ZADD ranking:game 1600 "player:ana"        # atualiza score da Ana

# Top 3 jogadores (maior score primeiro)
ZREVRANGE ranking:game 0 2 WITHSCORES
# "player:carol" 1800, "player:ana" 1600, "player:bob" 1200

# Posição de um jogador (0-indexed)
ZREVRANK ranking:game "player:ana"         # 1 (segunda posição)

# Quantos jogadores com score entre 1000 e 1500?
ZCOUNT ranking:game 1000 1500              # filtra por range de score

# Agendamento de tarefas (score = timestamp unix)
ZADD scheduler:jobs 1711310400 "job:enviar-email"
ZADD scheduler:jobs 1711314000 "job:gerar-relatorio"
# Buscar jobs prontos para executar (score <= agora)
ZRANGEBYSCORE scheduler:jobs 0 1711312000

Caso real: O Twitter Trends usa Sorted Sets para calcular trending topics. O score é a contagem de menções em janela de tempo. ZINCRBY incrementa atomicamente a cada tweet, e ZREVRANGE retorna os top N em tempo real.

Expiração de Chaves: TTL em Detalhe

Definir e consultar TTL

# TTL ao criar (segundos / ms)
SET session:abc '{"u":"Ana"}' EX 3600
SET temp:x "val" PX 500

# Adicionar TTL a chave existente
EXPIRE user:123:profile 600

# Consultar TTL restante
TTL session:abc    # -1=sem TTL, -2=não existe

# Remover TTL (fica permanente)
PERSIST session:abc

Comportamento

  • Chave sem TTL = permanente (até eviction ou DEL)
  • TTL retorna -1 (sem expiração) ou -2 (não existe)
  • Redis verifica expiração de forma lazy (no acesso) + ativa (sampling periódico)
  • EX = segundos, PX = milissegundos
  • PEXPIRE / PTTL para precisão em ms

Regra de ouro: SEMPRE defina TTL para chaves de cache. Chaves sem TTL acumulam silenciosamente e causam estouro de memória.

Caso Real: Cache Invalidation no Twitter

O Twitter enfrenta o desafio de manter timelines atualizadas em cache para milhões de usuários.

Problema: Como invalidar corretamente o cache da timeline ao publicar, deletar ou retweetar?

Solução: Uso de mecanismos de invalidação seletiva, como remoção ou atualização das chaves relacionadas, para evitar dados obsoletos.

Cache Invalidation é conhecido como um dos problemas mais difíceis em sistemas distribuídos.

"There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton

Exemplo Cache-Aside com Java/Spring Boot

// Método para obter usuário com cache Redis
public User getUserById(String id) {
  String cacheKey = "user:" + id;
  String cached = redisTemplate.opsForValue().get(cacheKey);
  if (cached != null) {
    return objectMapper.readValue(cached, User.class);
  }
  User user = userRepository.findById(id);
  redisTemplate.opsForValue().set(
    cacheKey, objectMapper.writeValueAsString(user), 10, TimeUnit.MINUTES
  );
  return user;
}

Exercício: Implementar Cache-Aside com Redis e Node.js

Objetivo: Criar uma API Node.js que consulta um banco simulado e utiliza Redis como cache seguindo o padrão Cache-Aside.

Passos:

  1. Subir Redis com Docker:
    docker run -d --name meu-redis -p 6379:6379 redis:7-alpine
  2. Configurar cliente Redis na aplicação Node.js (npm install redis express)
  3. Implementar endpoint GET /users/:id que:
    • Verifica cache Redis
    • Se não existir, consulta banco simulado (array ou JSON)
    • Armazena resultado no Redis com TTL (ex: 10 minutos)
    • Retorna o dado ao cliente com indicação da fonte (cache ou banco)
  4. Testar com diversas requisições e monitorar via redis-cli MONITOR

Exemplo de código Node.js para Cache-Aside

const express = require('express');
const redis = require('redis');
const app = express();
const redisClient = redis.createClient();

const users = [
  { id: '1', name: 'Ana' },
  { id: '2', name: 'Bruno' },
  { id: '3', name: 'Carla' }
];

redisClient.connect();

app.get('/users/:id', async (req, res) => {
  const id = req.params.id;
  const cacheKey = `user:${id}`;

  try {
    const cachedUser = await redisClient.get(cacheKey);
    if (cachedUser) {
      return res.json({ source: 'cache', data: JSON.parse(cachedUser) });
    }

    const user = users.find(u => u.id === id);
    if (!user) return res.status(404).json({ error: 'User not found' });

    await redisClient.set(cacheKey, JSON.stringify(user), { EX: 600 });
    res.json({ source: 'database', data: user });

  } catch (err) {
    res.status(500).json({ error: 'Internal server error' });
  }
});

app.listen(3000, () => console.log('API rodando na porta 3000'));

Resumo e Aprendizados

  • Cache reduz latência em ~100x comparado a queries de banco de dados
  • Redis: ~100k ops/s, latência < 1ms, single-threaded por design
  • Convenção de chaves: objeto:id:atributo com prefixo por serviço
  • SEMPRE definir TTL; usar allkeys-lru para cache puro
  • Docker: redis:7-alpine com --maxmemory para ambientes controlados
  • Cache Invalidation continua sendo um dos problemas mais difíceis em sistemas distribuídos
