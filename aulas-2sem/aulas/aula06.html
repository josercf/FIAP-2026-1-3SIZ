As estratégias de leitura (Cache-Aside, Read-Through) e escrita (Write-Through, Write-Behind) podem ser combinadas. Ex: Cache-Aside para leitura + Write-Through para escrita.

Cache-Aside (Lazy Loading)

VantagensDesvantagens
Só carrega no cache o que realmente é lido Primeiro request sempre tem latência total (cold start)
Cache falhar não derruba a aplicação Dados podem ficar stale entre TTL e atualização no banco

Caso real: Netflix usa Cache-Aside para catálogo de filmes. ~90% dos requests são servidos pelo cache. Dados atualizados a cada 15-30 min via TTL — aceitável porque metadados de filmes mudam raramente.

Read-Through

VantagensDesvantagens
Código da aplicação mais limpo — não gerencia cache Precisa de um cache provider que suporte loaders
Centraliza a lógica de carregamento no provider Debugging mais difícil (lógica escondida na camada de cache)

Na prática: bibliotecas como Caffeine (Java) e NCache (.NET) implementam Read-Through nativamente. Você configura um loader e a lib cuida do resto. Redis puro não suporta Read-Through — precisa de uma camada wrapper na aplicação.

Write-Through

VantagensDesvantagens
Cache sempre consistente com o banco Latência de escrita ~2x (escreve em dois lugares)
Leituras subsequentes sempre HIT Pode cachear dados que nunca serão lidos (desperdício de memória)

Caso real: DynamoDB Accelerator (DAX) da AWS usa Write-Through. Toda escrita no DynamoDB passa pelo DAX automaticamente, garantindo que leituras sempre retornem dados atualizados com latência de microsegundos.

Write-Behind (Write-Back)

VantagensDesvantagens
Latência de escrita mínima (só cache) Risco de perda de dados se o cache falhar antes do sync
Permite agrupar escritas em batch (reduz carga no banco) Complexidade de implementação (fila, retries, dead-letter)

Caso real: Sistemas de analytics e contadores (views, likes, page hits) usam Write-Behind. O Facebook acumula contadores de likes em memória e faz flush para o banco a cada poucos segundos — perder um ou dois likes é aceitável, mas a latência de escrita precisa ser mínima.

Redis: Políticas de Eviction (maxmemory-policy)

PolicyComportamento
noevictionRetorna erro quando memória cheia (padrão)
allkeys-lruRemove chave menos acessada recentemente (recomendado para cache)
volatile-lruLRU apenas em chaves com TTL definido
allkeys-lfuRemove chave com menor frequência de acesso
volatile-lfuLFU apenas em chaves com TTL definido
volatile-ttlRemove chave com menor TTL restante
 
# Configurar limite de memória e política
CONFIG SET maxmemory 256mb
CONFIG SET maxmemory-policy allkeys-lru

Regra prática: para cache puro, use allkeys-lru. Para mix de cache + dados persistentes, use volatile-lru (só remove chaves com TTL).

Redis como Cache Distribuído

  • Armazenamento chave-valor em memória, escrito em C
  • Suporta múltiplas estruturas: strings, hashes, listas, sets, sorted sets
  • Alta disponibilidade com replicação (master-replica) e persistência opcional
  • Suporte a clusters para escalabilidade horizontal (particionamento automático)
  • Ideal para cache distribuído em arquiteturas de microsserviços

Redis: Números de Performance

MétricaValor
Throughput (GET/SET)~100.000+ ops/s em instância única
Latência média< 1 ms para operações simples
Com pipeliningaté ~1.000.000 ops/s
Modelo de execuçãoSingle-threaded com I/O multiplexing (epoll/kqueue)
Capacidade por GB~15-17 milhões de chaves string simples
Tamanho máximo de chave512 MB
Tamanho máximo de valor512 MB

Por que single-threaded? Evita overhead de locks e context switching. O gargalo geralmente é a rede, não a CPU. Redis 7+ usa threads auxiliares para I/O de rede, mantendo a execução de comandos single-threaded.

Redis Keys: Padrão e Convenções

Formato recomendado: objeto:id:atributo
Exemplos: user:123:profile, session:abc-def, cache:product:456
Separador padrão: : (dois-pontos)
Convenção universal no ecossistema Redis. Facilita organização e busca.
Prefixos por serviço (microsserviços):
order-svc:order:789, auth-svc:session:abc — evita colisão entre serviços
Evitar:
Chaves > 1 KB, espaços, caracteres especiais. Chaves curtas = menos memória.
# Boas práticas de nomeação
SET user:123:name "João"
SET user:123:email "joao@email.com"
SET session:abc-def-ghi '{"userId":"123"}'
SET cache:product:456 '{"nome":"Notebook","preco":4500}'

Expiração de Chaves: TTL em Detalhe

# Definir TTL ao criar
SET session:abc '{"user":"Ana"}' EX 3600    # expira em 1 hora
SET temp:data "valor" PX 500                # expira em 500ms

# Adicionar TTL a chave existente
EXPIRE user:123:profile 600                 # 10 minutos
PEXPIRE user:123:profile 600000             # mesmo, em milissegundos

# Consultar TTL restante
TTL session:abc          # retorna segundos restantes (-1 = sem TTL, -2 = não existe)

# Remover TTL (chave fica permanente)
PERSIST session:abc
  • Chave sem TTL = permanente (até eviction ou DEL manual)
  • TTL retorna -1 se a chave existe sem expiração; -2 se não existe

Regra de ouro: SEMPRE defina TTL para chaves de cache. Chaves sem TTL acumulam silenciosamente e causam estouro de memória.

Caso Real: Cache Invalidation no Twitter

O Twitter enfrenta o desafio de manter timelines atualizadas em cache para milhões de usuários.

Problema: Como invalidar corretamente o cache da timeline ao publicar, deletar ou retweetar?

Solução: Uso de mecanismos de invalidação seletiva, como remoção ou atualização das chaves relacionadas, para evitar dados obsoletos.

Cache Invalidation é conhecido como um dos problemas mais difíceis em sistemas distribuídos.

"There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton

Exemplo Cache-Aside com Java/Spring Boot

// Método para obter usuário com cache Redis
public User getUserById(String id) {
  String cacheKey = "user:" + id;
  String cached = redisTemplate.opsForValue().get(cacheKey);
  if (cached != null) {
    return objectMapper.readValue(cached, User.class);
  }
  User user = userRepository.findById(id);
  redisTemplate.opsForValue().set(
    cacheKey, objectMapper.writeValueAsString(user), 10, TimeUnit.MINUTES
  );
  return user;
}

Exercício: Implementar Cache-Aside com Redis e Node.js

Objetivo: Criar uma API Node.js que consulta um banco simulado e utiliza Redis como cache seguindo o padrão Cache-Aside.

Passos:

  1. Subir Redis com Docker:
    docker run -d --name meu-redis -p 6379:6379 redis:7-alpine
  2. Configurar cliente Redis na aplicação Node.js (npm install redis express)
  3. Implementar endpoint GET /users/:id que:
    • Verifica cache Redis
    • Se não existir, consulta banco simulado (array ou JSON)
    • Armazena resultado no Redis com TTL (ex: 10 minutos)
    • Retorna o dado ao cliente com indicação da fonte (cache ou banco)
  4. Testar com diversas requisições e monitorar via redis-cli MONITOR

Exemplo de código Node.js para Cache-Aside

const express = require('express');
const redis = require('redis');
const app = express();
const redisClient = redis.createClient();

const users = [
  { id: '1', name: 'Ana' },
  { id: '2', name: 'Bruno' },
  { id: '3', name: 'Carla' }
];

redisClient.connect();

app.get('/users/:id', async (req, res) => {
  const id = req.params.id;
  const cacheKey = `user:${id}`;

  try {
    const cachedUser = await redisClient.get(cacheKey);
    if (cachedUser) {
      return res.json({ source: 'cache', data: JSON.parse(cachedUser) });
    }

    const user = users.find(u => u.id === id);
    if (!user) return res.status(404).json({ error: 'User not found' });

    await redisClient.set(cacheKey, JSON.stringify(user), { EX: 600 });
    res.json({ source: 'database', data: user });

  } catch (err) {
    res.status(500).json({ error: 'Internal server error' });
  }
});

app.listen(3000, () => console.log('API rodando na porta 3000'));

Resumo e Aprendizados

  • Cache reduz latência em ~100x comparado a queries de banco de dados
  • Redis: ~100k ops/s, latência < 1ms, single-threaded por design
  • Convenção de chaves: objeto:id:atributo com prefixo por serviço
  • SEMPRE definir TTL; usar allkeys-lru para cache puro
  • Docker: redis:7-alpine com --maxmemory para ambientes controlados
  • Cache Invalidation continua sendo um dos problemas mais difíceis em sistemas distribuídos
