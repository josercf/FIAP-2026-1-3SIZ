Database per Service Pattern

Cada microsserviço possui seu próprio banco de dados, garantindo independência e menor acoplamento.

  • Isolamento entre serviços
  • Escalabilidade independente
  • Flexibilidade na escolha da tecnologia de BD
  • Desafios: consistência e transações distribuídas

Shared Database

Anti-pattern e Quando Faz Sentido

Anti-pattern: vários serviços acessam o mesmo banco, causando alto acoplamento e riscos de inconsistência.

Quando pode ser usado:

  • Legado com tempo curto para migração
  • Ambientes controlados e pequenos
  • Quando a consistência forte é mandatória e difícil de se implementar de outra forma

Dois Modelos: Pub/Sub vs Event Streaming

Pub/Sub (Publish-Subscribe)

  • Mensagem entregue e removida da fila
  • Cada mensagem consumida por um consumidor do grupo
  • Ideal para tarefas (processar pagamento, enviar e-mail)
  • Ex: RabbitMQ, Amazon SQS, Azure Service Bus

Event Streaming

  • Eventos persistidos em um log ordenado
  • Múltiplos consumidores leem o mesmo stream
  • Permite replay: reprocessar eventos do passado
  • Ex: Apache Kafka, Amazon Kinesis, Azure Event Hubs

Na prática: muitas arquiteturas combinam ambos. Kafka para o stream principal de eventos, RabbitMQ/SQS para filas de trabalho específicas. O importante é escolher pelo requisito: precisa de replay e auditoria? Streaming. Precisa só de entrega confiável? Pub/Sub.

Saga Pattern

Consistência em Transações Distribuídas

Em microsserviços, cada serviço tem seu próprio banco — não há transação ACID global. Como manter consistência?

Saga é uma sequência de transações locais. Cada serviço executa sua operação e dispara a próxima etapa via evento ou mensagem. Se uma etapa falha, transações compensatórias desfazem o que já foi feito.

Saga: Transações Compensatórias

O "Ctrl+Z" dos Microsserviços

Cada etapa da saga precisa ter uma ação compensatória caso precise ser revertida.

EtapaTransaçãoCompensação
1. PedidoCriar pedidoCancelar pedido
2. PagamentoCobrar cartãoEstornar pagamento
3. EstoqueReservar itensLiberar reserva
4. EnvioAgendar entregaCancelar agendamento

Transação Pivot (ponto sem retorno): a partir dela, as próximas etapas devem ser retriable (idempotentes e com retry). Exemplo: após o pagamento confirmado, o envio deve ser garantido com retentativas.

Quando usar cada abordagem?

Use Coreografia quando...

  • Fluxo simples com poucos serviços (2-4)
  • Times diferentes mantêm cada serviço
  • Consistência eventual é aceitável
  • Precisa de alta resiliência e desacoplamento
  • Serviços já usam arquitetura orientada a eventos

Use Orquestração quando...

  • Fluxo complexo com muitos serviços (5+)
  • Precisa de controle centralizado e monitoramento
  • Lógica de compensação é complexa
  • Consistência e rastreabilidade são prioridade
  • Mesmo time mantém os serviços envolvidos

Abordagem híbrida: na prática, muitas empresas combinam as duas. Exemplo: orquestração para o fluxo principal de compra e coreografia para notificações e analytics.
