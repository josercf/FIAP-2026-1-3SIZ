O Problema: Comunicação Direta

Em uma arquitetura de microsserviços, o cliente precisa conhecer o endereço de cada serviço individualmente. Isso gera diversos problemas:

Acoplamento

O cliente precisa saber a localização exata de cada microsserviço

Cross-cutting concerns

Autenticação, logging e rate limiting precisam ser replicados em cada serviço

Segurança

Cada serviço fica exposto diretamente à internet

Evolução

Mudar a estrutura interna dos serviços impacta diretamente os clientes

O que é um API Gateway?

O API Gateway é um ponto de entrada único para todas as requisições dos clientes em uma arquitetura de microsserviços. Ele atua como um intermediário inteligente entre o mundo externo e os serviços internos.

Cliente
(Browser/App)
→
API Gateway
→
Serviço A
↘
Serviço B

Analogia: O API Gateway funciona como a recepção de um hotel — o hóspede não precisa saber onde fica a cozinha, a lavanderia ou a manutenção. Ele fala com a recepção e ela direciona o pedido.

Mas nem todo Gateway é igual!

Lembra do modelo OSI? Gateways podem atuar em camadas diferentes. Dependendo da camada, ele "enxerga" coisas diferentes da requisição.

Camada 7 – Aplicação

Entende URLs, headers HTTP, body, cookies. É aqui que o API Gateway atua.

Camadas 3/4 – Rede e Transporte

Enxerga apenas endereços IP e portas TCP. Atua como Load Balancer de rede.

Layer 3/4 vs Layer 7 — O que cada um "enxerga"?

Network Gateway (L3/4)

Olha apenas o "envelope" da requisição:

  • IP de origem e destino
  • Porta TCP (ex.: 80, 443, 8080)
  • Protocolo (TCP/UDP)

Analogia: É o porteiro do prédio — sabe de qual andar você veio e para qual vai, mas não lê suas cartas.

API Gateway (L7)

Abre e entende o conteúdo da requisição:

  • URL / path (/api/pedidos)
  • Método HTTP (GET, POST...)
  • Headers (Authorization, Content-Type)
  • Body (JSON, XML...)

Analogia: É a recepcionista — lê o pedido, verifica seu crachá e direciona para o setor certo.

Mini-mundo: Dois problemas, duas camadas

A Loja TechBR está crescendo. O time de infra precisa resolver dois problemas diferentes. Cada um exige um tipo de gateway:

Problema 1 — Camada 3/4 (IP/TCP)

A loja tem 3 servidores idênticos rodando a mesma aplicação. Quando chega muito tráfego, um servidor sozinho não aguenta.

Solução: Um Load Balancer L3/4 distribui as conexões pelos 3 servidores, olhando apenas o IP e a porta TCP. Não precisa saber o que tem dentro da requisição.

Clientes
→

:80
→
Srv 1
Srv 2
Srv 3

Problema 2 — Camada 7 (HTTP)

A loja agora tem microsserviços separados: Catálogo, Carrinho e Pagamento. O gateway precisa ler a URL para saber para onde enviar.

Solução: Um API Gateway L7 lê o path HTTP e roteia: /catalogo → Serviço Catálogo, /carrinho → Serviço Carrinho.

→
API GW
L7
→
/catalogo
/carrinho

Resumo: O API Gateway (L7) usa sessões da Camada 5 e se baseia na infraestrutura IP/TCP (L3/4) para fazer o gerenciamento avançado de APIs. São complementares, não concorrentes!

Proxy Reverso vs Proxy Direto

Proxy Direto (Forward Proxy)

Fica entre o cliente e a internet. O cliente sabe que está usando um proxy.

  • Anonimiza o cliente
  • Filtra conteúdo (firewall corporativo)
  • Cache de saída

Ex.: Proxy corporativo, VPN

Proxy Reverso (Reverse Proxy)

Fica entre a internet e os servidores. O cliente não sabe que existe um proxy.

  • Protege os servidores
  • Distribui carga (Load Balancing)
  • Terminação SSL

Ex.: Nginx, Traefik, HAProxy

Nginx como Proxy Reverso

O Nginx é um dos servidores web e proxy reverso mais utilizados no mundo. Sua arquitetura orientada a eventos o torna extremamente eficiente para lidar com milhares de conexões simultâneas.

# nginx.conf - Exemplo de Proxy Reverso
upstream api_pedidos {
    server pedidos-service:8080;
}

upstream api_produtos {
    server produtos-service:8081;
}

server {
    listen 80;
    server_name api.meusite.com;

    location /api/pedidos {
        proxy_pass http://api_pedidos;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    location /api/produtos {
        proxy_pass http://api_produtos;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

Traefik: Proxy Reverso Cloud-Native

O Traefik é um proxy reverso moderno que se integra nativamente com Docker, Kubernetes e outros orquestradores. Ele descobre serviços automaticamente — sem necessidade de configuração manual de rotas.

Vantagens

  • Auto-discovery de serviços via Docker labels
  • Certificados SSL automáticos (Let's Encrypt)
  • Dashboard de monitoramento integrado
  • Middlewares configuráveis (auth, rate limit, headers)

docker-compose.yml

services:
  traefik:
    image: traefik:v3.0
    ports:
      - "80:80"
      - "8080:8080"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    command:
      - "--providers.docker=true"
      - "--api.insecure=true"

  api-pedidos:
    image: minha-api:latest
    labels:
      - "traefik.http.routers.pedidos.rule=PathPrefix(`/api/pedidos`)"

Caso Real: Netflix Zuul → Spring Cloud Gateway

A Netflix foi pioneira no uso de API Gateways com o Zuul, que processava bilhões de requisições por dia. Em 2018, migraram para o Zuul 2 (assíncrono) e a comunidade Spring criou o Spring Cloud Gateway como alternativa reativa.

Problema real: Com mais de 1.000 microsserviços, a Netflix precisava de um ponto central para roteamento, autenticação, canary deployments e resiliência. Sem um gateway, cada serviço teria que implementar tudo isso individualmente.

Canary Deployment

O gateway direciona 5% do tráfego para a nova versão antes do rollout completo

A/B Testing

Diferentes versões da API servidas para diferentes grupos de usuários

Hands-on: Nginx como Proxy Reverso com Docker

🎯 Objetivo

Configurar o Nginx como proxy reverso para rotear requisições para duas APIs diferentes rodando em containers Docker.

Repositório do Exercício github.com/josercf/2026-1-1002-aula01

  1. Clone o repositório: git clone https://github.com/josercf/2026-1-1002-aula01.git
  2. Configure dois serviços simples usando a imagem hashicorp/http-echo: um retornando "Serviço A" na porta 5001 e outro "Serviço B" na porta 5002
  3. Crie o arquivo nginx.conf com rotas: /servico-a → container A e /servico-b → container B
  4. Adicione o Nginx como serviço no docker-compose, mapeando a porta 80 e montando o nginx.conf
  5. Execute docker compose up -d e teste com curl http://localhost/servico-a
  6. Desafio: Adicione um header customizado X-Gateway: fiap-proxy nas respostas
01 – Revisão + API Gateway & Proxy Reverso
21

Solução: docker-compose.yml

version: '3.8'
services:
  servico-a:
    image: hashicorp/http-echo
    command: ["-text=Resposta do Serviço A", "-listen=:5001"]
    networks:
      - gateway-net

  servico-b:
    image: hashicorp/http-echo
    command: ["-text=Resposta do Serviço B", "-listen=:5002"]
    networks:
      - gateway-net

  nginx-gateway:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - servico-a
      - servico-b
    networks:
      - gateway-net

networks:
  gateway-net:
    driver: bridge

Solução: nginx.conf

upstream servico_a {
    server servico-a:5001;
}

upstream servico_b {
    server servico-b:5002;
}

server {
    listen 80;

    # Header customizado em todas as respostas (desafio)
    add_header X-Gateway "fiap-proxy" always;

    location /servico-a {
        proxy_pass http://servico_a/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    location /servico-b {
        proxy_pass http://servico_b/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

Teste: curl -i http://localhost/servico-a — observe o header X-Gateway: fiap-proxy na resposta!

Quiz

Qual é a principal diferença entre um Proxy Direto (Forward Proxy) e um Proxy Reverso (Reverse Proxy)?
  • A O Proxy Direto é mais rápido que o Proxy Reverso
  • B O Proxy Reverso só funciona com HTTPS, enquanto o Direto funciona com HTTP
  • C O Proxy Direto protege o cliente, enquanto o Proxy Reverso protege os servidores
  • D Não há diferença — são apenas nomes diferentes para a mesma tecnologia
