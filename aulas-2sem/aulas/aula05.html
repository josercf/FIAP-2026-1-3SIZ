Caso Real: Noisy Neighbor em aplicações multi-tenant

Noisy Neighbor: quando um tenant consome recursos excessivamente, impactando os demais.

Salesforce usa estratégias avançadas de particionamento para isolar tenants:

  • Isolamento por banco de dados dedicado ou schemas separados
  • Rate limiting e QoS para limitar impacto
  • Monitoramento contínuo e auto scaling
  • Uso de filas e caching para desacoplar cargas

Lab: Arquitetura Orientada a Eventos na Prática

Vamos explorar uma aplicação EDA funcional com RabbitMQ, Node.js e Docker — praticando scaling, falhas e acúmulo de mensagens.

O que vamos praticar:

  1. Exploração: Criar pedidos e observar o fanout para 4 consumers no dashboard em tempo real
  2. Scaling: Escalar consumers para aumentar throughput (competing consumers)
  3. Falhas: Parar serviços, ver mensagens acumulando na fila, reiniciar e observar o drain
  4. Reflexão: Comparar EDA vs REST síncrono, trade-offs e quando usar cada um

Repositório: github.com/josercf/eda-demo

Arquitetura da Demo

  • 1 mensagem publicada → 4 filas recebem (fanout exchange)
  • Cada consumer tem fila dedicada e delay diferente (simula carga real)
  • Qualquer consumer pode ser escalado independentemente

Exercício 1: Scaling Horizontal

O serviço de pagamento é o mais lento (2s por mensagem). Vamos escalar:


# Escalar pagamento para 5 instâncias
docker compose up -d --scale pagamento=5

# Verificar
docker compose ps | grep pagamento

# Flood de 100 pedidos pelo Dashboard e observar:
# - Throughput ~5x maior no pagamento
# - Round-robin entre as 5 instâncias
# - RabbitMQ: fila.pagamento com 5 consumers conectados

Competing Consumers: Múltiplas instâncias compartilham a mesma fila. O RabbitMQ distribui as mensagens em round-robin com prefetch(1) para distribuição justa.

Exercício 2: Falhas e Acúmulo de Mensagens


# 1. Parar o serviço de pagamento
docker compose stop pagamento

# 2. Enviar 100 pedidos pelo Dashboard (Flood)

# 3. Observar no RabbitMQ Management:
#    - fila.pagamento: mensagens acumulando (Ready)
#    - Outras filas: processando normalmente!

# 4. Reiniciar e ver o drain
docker compose start pagamento
# → Mensagens acumuladas são processadas na ordem (FIFO)

Desacoplamento total: O pagamento ficou offline mas estoque, notificação e analytics continuaram funcionando. Em REST síncrono, a falha propagaria para o chamador.

Reflexão: EDA vs REST Síncrono

Vantagens EDA

  • Desacoplamento total entre serviços
  • Escalabilidade independente por consumer
  • Resiliência a falhas (fila absorve)
  • Adicionar consumers sem alterar o producer

Trade-offs

  • Consistência eventual (não imediata)
  • Debugging e rastreamento mais complexo
  • Infraestrutura adicional (broker)
  • Ordenação de mensagens nem sempre garantida

Desafio: Projetar a Arquitetura de um Sistema

Formem duplas ou trios e escolham UM cenário:

  1. Fintech de Pagamentos: Processar 10k transações/min com auditoria completa e SLA 99.99%
  2. Plataforma de Streaming: Catálogo de 50k filmes, recomendações personalizadas e 500k usuários simultâneos
  3. Sistema Hospitalar: Prontuário eletrônico multi-unidade, agendamento e notificações, com dados sensíveis (LGPD)

Entregas (50 min):

  1. Definir a arquitetura: Quais padrões usar? (Database per Service, Saga, CQRS, Event Sourcing, EDA)
  2. Desenhar no Draw.io: Diagrama com serviços, bancos, filas e fluxo de dados
  3. Escolher serviços de nuvem: AWS, Azure ou GCP — justifiquem cada escolha
  4. Calcular o SLA composto: Com base nos SLAs dos serviços escolhidos
