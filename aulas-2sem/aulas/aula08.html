ISO/IEC 25010 – Qualidade de Produto de Software

Norma internacional que define o modelo de qualidade de software. TDD contribui diretamente para várias características:

Confiabilidade

Maturidade, disponibilidade, tolerância a falhas. TDD detecta bugs antes de chegarem à produção.

Manutenibilidade

Modularidade, reusabilidade, testabilidade. TDD força código desacoplado e coeso.

Adequação Funcional

Completude, correção. Os testes documentam o comportamento esperado.

Segurança

Confidencialidade, integridade. Testes de borda previnem falhas de segurança.

Portabilidade

Adaptabilidade, instalabilidade. Suíte de testes valida comportamento em novos ambientes.

Desempenho

Comportamento temporal, utilização de recursos. Testes de performance integram a suite.

O Problema: Código sem Testes

Sintomas comuns

  • Medo de refatorar — "se funciona, não mexa"
  • Regressões frequentes em produção
  • Bugs encontrados pelo cliente, não pela equipe
  • Integração entre times quebra silenciosamente
  • Onboarding lento: "rodou aqui, não sei por quê"

Com TDD

  • Confiança para refatorar — testes provam que nada quebrou
  • Feedback imediato a cada mudança
  • Testes são documentação executável do comportamento
  • Design naturalmente desacoplado (testável)
  • Redução do custo de bug: detectar cedo é barato
"Toda vez que você sente tentação de escrever um print para testar algo, escreva um teste em vez disso." — Kent Beck

O Ciclo TDD

🔴 RED

Escreva um teste para o comportamento que você quer implementar.

O teste deve falhar — o código ainda não existe.

🟢 GREEN

Implemente o mínimo possível para o teste passar.

Não se preocupe com elegância ainda — apenas faça passar.

🔵 REFACTOR

Melhore o código sem alterar o comportamento.

Os testes garantem que você não quebrou nada.

As duas regras de Kent Beck: (1) só escreva código para fazer um teste falho passar • (2) elimine toda duplicação

Padrão AAA – Arrange, Act, Assert

Toda boa estrutura de teste segue três etapas:


[Fact]
public void Divisao_PorZero_LancaExcecao()
{
    // ARRANGE – monte o cenário
    var calculadora = new Calculadora();

    // ACT – execute a ação
    Action acao = () => calculadora.Divisao(10, 0);

    // ASSERT – verifique o resultado
    acao.Should()
        .Throw<DivideByZeroException>()
        .WithMessage("Divisão por zero não é permitida.");
}

SOLID – Os 5 Princípios

S — Single Responsibility

Cada classe tem uma única razão para mudar. Facilita o teste isolado de responsabilidades.

O — Open / Closed

Aberta para extensão, fechada para modificação. Adicione comportamento sem alterar código existente.

L — Liskov Substitution

Subtipos devem ser substituíveis pelos seus tipos base sem quebrar o comportamento.

I — Interface Segregation

Interfaces pequenas e coesas. Clientes não devem depender de métodos que não usam.

D — Dependency Inversion ⭐

Dependa de abstrações, não de implementações concretas. É o princípio que habilita o TDD!

DIP + Interfaces + Injeção de Dependência = código desacoplado e testável

Framework de testes. [Fact] para casos únicos, [Theory] + [InlineData] para testes parametrizados.

O Problema: Código Acoplado


// ❌ Código acoplado — impossível testar sem enviar e-mail de verdade
public class PedidoService
{
    public void Confirmar(Pedido pedido)
    {
        // Regra de negócio misturada com infraestrutura
        var smtp = new SmtpClient("smtp.empresa.com"); // criação direta!
        smtp.Send("cliente@email.com", "Pedido confirmado!");
    }
}

Problemas

Não dá para testar sem servidor SMTP • Teste lento e frágil • Viola SRP e DIP (SOLID) • Impossível substituir por outro provedor

Solução: Interfaces + Inversão de Controle


// ✅ Depende da abstração, não da implementação concreta
public interface INotificacaoService
{
    void Notificar(string destinatario, string mensagem);
}

public class PedidoService
{
    private readonly INotificacaoService _notificacao;

    // Injeção de Dependência via construtor
    public PedidoService(INotificacaoService notificacao)
        => _notificacao = notificacao;

    public void Confirmar(Pedido pedido)
    {
        // Regra de negócio pura — sem infraestrutura aqui
        _notificacao.Notificar(pedido.ClienteEmail, "Pedido confirmado!");
    }
}

IoC (Inversão de Controle): o controle de qual implementação usar sai da classe e vai para quem a constrói (framework DI, teste, main).

DI no Teste: Substituindo Dependências


// ✅ No teste, injetamos o que quisermos — sem SMTP real!
[Fact]
public void Confirmar_PedidoValido_NotificaCliente()
{
    // Arrange
    var mockNotificacao = new Mock<INotificacaoService>();
    var service = new PedidoService(mockNotificacao.Object); // <-- injeção!
    var pedido = new Pedido("cliente@email.com");

    // Act
    service.Confirmar(pedido);

    // Assert — verificar se a notificação foi enviada
    mockNotificacao.Verify(
        n => n.Notificar("cliente@email.com", "Pedido confirmado!"),
        Times.Once
    );
}

// ❌ Como simular uma transação suspeita? Como testar offline?

Stub — Controla Entradas

Stub retorna valores pré-definidos. Você não verifica como foi chamado, só configura o que retorna.


// Cenário: PagamentoService consulta IDetectorFraude antes de processar
[Fact]
public void Pagar_TransacaoLimpa_ProcessaGateway()
{
    // Arrange — STUB: configura o retorno sem verificar chamadas
    var stubFraude = new Mock<IDetectorFraude>();
    stubFraude.Setup(f => f.EhSuspeito(It.IsAny<string>(), It.IsAny<decimal>()))
              .Returns(false); // <--- retorno fixo: nunca é fraude

    var stubGateway = new Mock<IGatewayPagamento>();
    stubGateway.Setup(g => g.Processar(It.IsAny<string>(), It.IsAny<decimal>()))
               .Returns(new ResultadoPagamento(true));

    var service = new PagamentoService(stubGateway.Object, stubFraude.Object);

    // Act
    var resultado = service.Pagar("4111-1111-1111-1111", 100m);

    // Assert — foco no resultado, não nas interações
    resultado.Sucesso.Should().BeTrue();
}

O Problema: Verificar Efeitos Colaterais


// O resultado diz "bloqueado"... mas o gateway foi chamado mesmo assim?
// Stub não nos conta — apenas retorna o valor configurado.
var resultado = service.Pagar("cartao-suspeito", 9999m);
resultado.Sucesso.Should().BeFalse(); // OK... mas e o gateway?

Problema

Stub controla retornos, mas não verifica se o método foi chamado, com quais argumentos, e quantas vezes. Efeitos colaterais passam despercebidos.

Solução: Mock

Além de configurar retornos, Mock registra todas as chamadas e permite verificar: "foi chamado exatamente uma vez com esse argumento?"

Mock — Verifica Interações

Mock vai além do Stub: você verifica se determinados métodos foram chamados, com quais argumentos e quantas vezes.


[Fact]
public void Pagar_TransacaoSuspeita_NuncaChamaGateway()
{
    // Arrange
    var mockFraude = new Mock<IDetectorFraude>();
    mockFraude.Setup(f => f.EhSuspeito(It.IsAny<string>(), It.IsAny<decimal>()))
              .Returns(true); // transação suspeita

    var mockGateway = new Mock<IGatewayPagamento>();
    var service = new PagamentoService(mockGateway.Object, mockFraude.Object);

    // Act
    service.Pagar("4111-1111-1111-1111", 100m);

    // Assert — MOCK: verifica que o gateway NUNCA foi chamado
    mockGateway.Verify(
        g => g.Processar(It.IsAny<string>(), It.IsAny<decimal>()),
        Times.Never  // <--- verificação de interação!
    );
}

Fake — Implementação Simplificada

Fake é uma implementação real mas simplificada — ex: banco em memória em vez de PostgreSQL.


// Fake: repositório em memória (sem banco de dados)
public class FakePedidoRepository : IPedidoRepository
{
    private readonly List<Pedido> _pedidos = new();

    public void Salvar(Pedido pedido) => _pedidos.Add(pedido);

    public Pedido? BuscarPorId(int id)
        => _pedidos.FirstOrDefault(p => p.Id == id);
}

// No teste: usa o Fake em vez do banco real
[Fact]
public void BuscarPedido_PedidoExistente_RetornaPedido()
{
    var repo = new FakePedidoRepository(); // <-- fake, não SQL!
    repo.Salvar(new Pedido { Id = 1, Total = 500m });

    var resultado = repo.BuscarPorId(1);

    resultado.Should().NotBeNull();
    resultado!.Total.Should().Be(500m);
}

Resumo: Stub vs Mock vs Fake

TipoO que fazVerifica interações?Quando usar
Stub Retorna dados fixos Não Controlar entradas do sistema em teste
Mock Retorna dados + verifica chamadas Sim Garantir que efeitos colaterais aconteceram
Fake Implementação leve e funcional Não normalmente Banco em memória, APIs internas, repositórios

Dica prática: Em C#, Moq cria tanto Stubs quanto Mocks. Use .Setup() para Stub e adicione .Verify() para transformar em Mock.

Dica: Algoritmo CPF

1. Remove não numéricos
2. Verifica comprimento = 11
3. Rejeita todos dígitos iguais
4. Calcula 1º dígito verificador
   soma = Σ dígito[i] × (10 − i), i=0..8
   resto = soma % 11
   d1 = (resto < 2) ? 0 : 11 − resto
5. Repete para 2º dígito (pesos 11..2)

Debrief — N4: TDD com Interfaces e Moq

O que aconteceu no processo

  • Interfaces IGatewayPagamento e IDetectorFraude tornaram o código testável

TDD contribui para confiabilidade, manutenibilidade e adequação funcional — pilares da qualidade.

Red → Green → Refactor

Teste falha → implementação mínima → melhoria. Nunca escreva código sem um teste vermelho antes.

Interfaces + DI

IoC desacopla responsabilidades. DI via construtor é o padrão para código testável.

Mock / Stub / Fake

Stub controla entradas. Mock verifica interações. Fake é implementação leve funcional.
