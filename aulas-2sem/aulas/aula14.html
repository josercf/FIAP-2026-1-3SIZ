Docker Swarm

Visão Geral

Docker Swarm é a solução nativa do Docker para orquestração de containers, possibilitando a criação, gerenciamento e escalonamento de clusters Docker.

  • Services: Definem a aplicação e o conjunto de containers que serão executados.
  • Replicas: Número de instâncias do serviço em execução para garantir disponibilidade.
  • Rolling Updates: Atualizações contínuas sem downtime, substituindo containers gradualmente.
  • Overlay Networks: Redes virtuais que interligam containers em diferentes hosts.

Docker Swarm

Exemplo: Criar serviço com 3 réplicas

docker service create --name webapp --replicas 3 -p 80:80 nginx:latest

Este comando cria um serviço chamado webapp com 3 réplicas do container Nginx, expondo a porta 80.

Docker Swarm

Rolling Update

Atualização gradual do serviço para uma nova imagem sem downtime.

docker service update --image nginx:1.21 webapp

O Swarm atualiza as réplicas uma a uma, garantindo alta disponibilidade do serviço.

Introdução a Kubernetes

Componentes básicos

  • Pods: Menor unidade de deploy, contém um ou mais containers que compartilham recursos.
  • Services: Abstração para expor Pods como serviços estáveis e acessíveis.
  • Deployments: Controlam atualizações e escalabilidade dos Pods.
  • ConfigMaps: Armazenam dados de configuração não sensíveis para os containers.
  • Secrets: Armazenam dados sensíveis, como senhas e tokens, de forma segura.

Kubernetes

Deployment e Rolling Update

Deployment gerencia réplicas e atualizações automáticas de Pods.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: meu-app
  template:
    metadata:
      labels:
        app: meu-app
    spec:
      containers:
      - name: app
        image: meuapp:v1
        ports:
        - containerPort: 8080

Para atualizar, basta alterar a imagem e aplicar o YAML, Kubernetes fará rolling update.

Kubernetes

ConfigMaps e Secrets

Permitem separar configurações e dados sensíveis do container.

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "DEBUG"
---
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DB_PASSWORD: c2VuaGExMjM=  # base64 encoded senha123

Pod pode consumir ConfigMaps e Secrets via variáveis de ambiente ou volumes.

Helm Charts

Gerenciamento simplificado de aplicações Kubernetes

Helm é um gerenciador de pacotes para Kubernetes que facilita deploy, upgrade e rollback de aplicações.

  • Chart: Pacote que contém templates e configurações para deployar uma aplicação.
  • Values.yaml: Arquivo para parametrizar o Chart.
  • Releases: Instâncias deployadas de um Chart.
helm install meu-app ./chart-meu-app
helm upgrade meu-app ./chart-meu-app

Caso Real: Globo.com

Migração de monolito para Kubernetes

A Globo.com enfrentou desafios comuns na migração de sua plataforma monolítica para arquitetura baseada em Kubernetes:

  • Desafios: Reestruturação da aplicação, adaptação de pipelines CI/CD, e gerenciamento do estado da aplicação.
  • Lições aprendidas: Importância do uso de ConfigMaps e Secrets, monitoramento ativo, e automação com Helm Charts.
  • Resultado: Melhor escalabilidade, resiliência e agilidade no desenvolvimento.

Exercício Prático

Deploy de aplicação com Docker Swarm

Objetivos:

  • Criar um serviço com 3 réplicas.
  • Executar um rolling update para atualizar a imagem do serviço.
  • Testar a alta disponibilidade do serviço durante o update.

Passos:

  1. Inicie o swarm (se ainda não iniciado): docker swarm init
  2. Crie o serviço com 3 réplicas usando a imagem nginx:latest:
    3. docker service create --name exercicio-webapp --replicas 3 -p 8080:80 nginx:latest
  3. Verifique o status do serviço e réplicas:
    4. docker service ls
docker service ps exercicio-webapp
  4. Atualize o serviço para a imagem nginx:1.21 com rolling update:
    5. docker service update --image nginx:1.21 exercicio-webapp
  5. Testar a disponibilidade acessando http://localhost:8080 durante a atualização.
  6. Repita a verificação do status do serviço para confirmar a atualização:
    7. docker service ps exercicio-webapp
  7. Finalize o exercício removendo o serviço:
    8. docker service rm exercicio-webapp

Quiz

Kubernetes Pods vs Containers

Qual das alternativas melhor descreve a relação entre Pods e Containers no Kubernetes?

  • Um Pod é um container isolado que roda sozinho.
  • Um Pod pode conter um ou mais containers que compartilham recursos.
  • Containers são agrupados em Services, e Pods são unidades lógicas.
  • Pods e Containers são sinônimos no Kubernetes.

Quiz

Quando usar Docker Swarm vs Kubernetes?

Qual cenário é mais adequado para escolher Docker Swarm em vez de Kubernetes?

  • Quando precisa de orquestração complexa e alta escalabilidade em multinuvem.
  • Para clusters simples com menor curva de aprendizado e integração nativa com Docker.
  • Quando é necessário gerenciamento sofisticado de volumes e rede nativa do Kubernetes.
  • Se a organização já usa Helm Charts amplamente.

Exemplo Java/Spring Boot

Deployment em Kubernetes

apiVersion: apps/v1
kind: Deployment
metadata:
  name: springboot-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: springboot
  template:
    metadata:
      labels:
        app: springboot
    spec:
      containers:
      - name: springboot
        image: minha-imagem/springboot-app:1.0
        ports:
        - containerPort: 8080
        envFrom:
        - configMapRef:
            name: springboot-config
        - secretRef:
            name: springboot-secret

Exemplo .NET / C#

Dockerfile e Deployment básico

FROM mcr.microsoft.com/dotnet/aspnet:6.0 AS base
WORKDIR /app
EXPOSE 80

FROM mcr.microsoft.com/dotnet/sdk:6.0 AS build
WORKDIR /src
COPY ["MyApp/MyApp.csproj", "MyApp/"]
RUN dotnet restore "MyApp/MyApp.csproj"
COPY . .
WORKDIR "/src/MyApp"
RUN dotnet build "MyApp.csproj" -c Release -o /app/build

FROM build AS publish
RUN dotnet publish "MyApp.csproj" -c Release -o /app/publish

FROM base AS final
WORKDIR /app
COPY --from=publish /app/publish .
ENTRYPOINT ["dotnet", "MyApp.dll"]

apiVersion: apps/v1
kind: Deployment
metadata:
  name: dotnet-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: dotnetapp
  template:
    metadata:
      labels:
        app: dotnetapp
    spec:
      containers:
      - name: dotnet
        image: minha-imagem/dotnet-app:latest
        ports:
        - containerPort: 80
